# MVP 范围定义

> 定义日期: 2026-04-24
> 目标: 明确第一版的精确范围，为架构选型和 PoC 验证提供基准参考

---

## 一、产品定位

一个支持大规模数据管理的多维表格，核心能力对标飞书多维表格。V1 聚焦**数据管理和表格视图**，后续版本扩展视图类型和高级功能。

## 二、V1 范围

### 2.1 核心数据能力

| 维度 | V1 目标 | 说明 |
|------|---------|------|
| 单表最大行数 | 10 万行（初期）/ 百万行（架构预留） | V1 以 10 万行为实际验证目标，但架构不做硬限制 |
| 单表最大列数 | 50 列 | 涵盖绝大多数业务场景 |
| 最大并发编辑 | 50 人同时编辑同一张表 | 不是 1000 人级别，但需要基本的多人协作能力 |
| 总表数 | 无限制（每 Space 建议 < 100 张） | 依赖 PostgreSQL，表数量不是瓶颈 |
| 附件存储 | 单文件 < 50MB | 初期控制存储成本 |

### 2.2 首批字段类型

| 字段类型 | 优先级 | 说明 |
|---------|--------|------|
| 文本 (Text) | P0 | 单行文本，支持富文本（可选） |
| 数字 (Number) | P0 | 整数/小数，支持格式化 |
| 日期 (Date) | P0 | 含日期和时间，支持时区 |
| 选择 (Select) | P0 | 单选，预定义选项列表 |
| 多选 (MultiSelect) | P0 | 多选，预定义选项列表 |
| 复选框 (Checkbox) | P1 | 布尔值 |
| 图片 (Image) | P1 | 支持上传和 URL 引用，S3 存储 |
| 关联 (Link/Relation) | P1 | 关联到其他表的行，单向关联优先 |
| 公式 (Formula) | P1 | 基于其他字段计算，首批支持 SUM/COUNT/IF 等基础函数 |
| 附件 (Attachment) | P2 | 上传文件，S3 存储 |
| 用户 (User) | P2 | 引用系统用户 |
| 创建时间/修改时间 | P1 | 自动记录时间戳 |

### 2.3 首批视图类型

| 视图类型 | 优先级 | 说明 |
|---------|--------|------|
| 表格视图 (Grid) | P0 | 核心视图，包含排序、筛选、分组、列隐藏/固定 |
| 表单视图 (Form) | P1 | 数据录入，支持字段配置和必填校验 |
| 看板视图 (Kanban) | P2 | 基于选择字段分组，卡片拖拽 |

### 2.4 核心功能

| 功能 | 优先级 | 说明 |
|------|--------|------|
| **行操作**：增/删/复制/拖拽排序 | P0 | 基础数据操作 |
| **列操作**：增/删/隐藏/排序/固定/调整宽度 | P0 | 列管理 |
| **筛选**：多条件 AND/OR 筛选 | P0 | 含按字段类型智能筛选 |
| **排序**：多列排序 | P0 |  |
| **分组**：最多 3 层分组 | P1 |  |
| **搜索**：全表文本搜索 | P1 |  |
| **撤销/重做** (Undo/Redo) | P1 | 本地操作回退 |
| **导入/导出**：CSV/Excel 导入，CSV 导出 | P1 |  |
| **权限**：表级别读/写权限 | P1 | 行/列级别权限 V2 |
| **用户系统**：注册/登录/JWT 认证 | P0 |  |
| **工作区**：支持多 Space，每 Space 含多张表 | P0 |  |

### 2.5 V1 明确不做

| 功能 | 原因 |
|------|------|
| 甘特图/日历/画廊/时间线视图 | 表格视图 + 表单视图覆盖核心场景 |
| 行级/列级细粒度权限 | 复杂度高，表级权限可满足初期需求 |
| 自动化工作流 (Trigger-Action) | 先做数据管理，再做自动化 |
| AI 集成 | V2 考虑 |
| 仪表盘/BI 图表 | V2 考虑 |
| 插件/Widget 系统 | V3 考虑 |
| 移动端原生 App | 响应式 Web 即可 |
| 第三方 OAuth 登录 | V2 考虑 |
| 国际化 (i18n) | 初期中文，代码层面预留 i18n 接口 |
| 模板市场 | V2 考虑 |
| 跨表公式引用 (Cross-Table Reference) | V2 考虑，V1 仅支持同表公式 |

## 三、技术约束

### 3.1 部署约束

| 维度 | V1 要求 |
|------|--------|
| 部署方式 | Docker Compose 单机部署（< 5 个容器） |
| 操作系统 | Linux (Ubuntu 22.04+) |
| 最低配置 | 4C8G + 100GB SSD |
| 云服务 | 不依赖特定云厂商，私有化可部署 |

### 3.2 性能目标

| 指标 | V1 目标 | 测试条件 |
|------|--------|---------|
| 表格首屏加载 | < 2 秒 | 10 万行 × 50 列，视窗内可见数据 |
| 表格滚动帧率 | ≥ 50 FPS | 大数据量下流畅滚动 |
| 单元格编辑延迟 | < 100ms | 本地操作，到 UI 更新 |
| API 响应延迟 (P99) | < 100ms | 简单 CRUD，不含搜索 |
| 搜索响应延迟 | < 1 秒 | 10 万行全表搜索 |
| 多客户端同步延迟 | < 500ms | 同城网络，3 客户端 |

### 3.3 技术原则

1. **先简单后复杂**：能用 PostgreSQL 解决的不引入新组件，能用乐观锁的不引入 CRDT
2. **PoC 验证后决策**：关键技术选型（Canvas vs DOM、CRDT vs OT、Rust vs TS）必须有 PoC 数据支撑
3. **架构预留，MVP 简化**：架构设计为百万行预留扩展点，但 MVP 实现可以有简化假设
4. **Docker 一键部署**：降低私有化部署门槛

## 四、V1 不做但架构需预留的能力

| 能力 | 预留方式 |
|------|---------|
| 百万行扩展 | 表支持分片 key 字段（`shard_key`），数据层接口支持分片路由 |
| CRDT 替换 | 单元格编辑抽象为 `applyOperation(op)` 接口，初始实现用乐观锁 |
| 更多视图类型 | 视图用多态设计，每种视图实现统一的 `render`/`applyFilter`/`applySort` 接口 |
| 公式引擎替换 | 公式计算通过 `formulaEval(formula, context) -> value` 抽象接口 |
| 行列级权限 | 数据访问通过 `canRead(cell) -> bool` 抽象，初始实现返回 true |

## 五、可参考的开源方案对照

| 功能 | 可参考的开源实现 |
|------|-----------------|
| Canvas 表格渲染 | APITable packages/grid (Konva.js) |
| 数据模型 | APITable Snapshot+Changeset 或 Teable PostgreSQL 直连 |
| 字段类型定义 | APITable field types 或 AppFlowy FieldType 枚举 |
| 公式引擎 | HyperFormula (JS) 或 Formualizer (Wasm) |
| 实时协作 | Yjs + y-websocket（简单场景） 或 APITable OT（复杂场景） |
| 导入/导出 | exceljs + csv-parser |
| 权限 | APITable Mirror 模式 或 PostgreSQL RLS |
