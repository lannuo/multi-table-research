## 一、整体架构

### 1.1 技术栈总览

飞书多维表格的底层是一个**自研数据库系统**，而非简单的 "PostgreSQL + 缓存" 架构：

```
┌── 应用层 ────────────────────────────────────────┐
│  多维表格视图 / 仪表盘 / 工作流 / AI 能力         │
├── 查询层 ────────────────────────────────────────┤
│  WideQuery — MPP 大规模并行查询引擎               │
│  参考: Apache Calcite, Greenplum, Presto, SparkSQL│
├── 存储层 ────────────────────────────────────────┤
│  FxDB — 领域专用分布式数据库 (自研, C/C++/Go)    │
│  参考: RocksDB, LevelDB, MySQL, Cassandra        │
├── 基础设施 ──────────────────────────────────────┤
│  Rust 公式引擎 / OT+CRDT 协作引擎 / Canvas 渲染  │
│  PostgreSQL (关系型元数据) / Redis (缓存)         │
└──────────────────────────────────────────────────┘
```

**关键基础设施团队**: CDI (Common Data Infra)，字节跳动内部数据库基础设施团队，负责 FxDB 和 WideQuery 的研发。

### 1.2 语言分布

| 语言 | 用途 |
|------|------|
| **Rust** | 公式引擎、Lark SDK 核心逻辑、WASM SQLite |
| **C/C++/Go** | FxDB 分布式存储引擎、WideQuery MPP |
| **TypeScript/JS** | Web 前端、微前端架构 |
| **Swift/Kotlin** | 移动端 |
| **Python?** | AI 相关 (未公开) |

## 二、自研数据库：FxDB + WideQuery

### 2.1 FxDB（领域专用分布式数据库）

这是飞书多维表格最核心的基础设施，为多维表格场景定制开发：

- **分布式架构**：支持分布式一致性协议、分布式事务
- **存储引擎参考**：RocksDB (LSM-tree)、LevelDB、MySQL、Cassandra
- **实现语言**：C/C++/Go
- **热行（Hot Row）存储**：每行承载动态计算值（公式、引用、跨表计算），与静态数据行区分
- **未开源**

### 2.2 WideQuery（MPP 大规模并行查询引擎）

- **MPP 架构**：大规模并行处理，对标 Greenplum/Presto/SparkSQL
- **查询优化器**：参考 Apache Calcite 的代价优化模型
- **分布式执行器**：跨节点并行查询，支持 200 张表联合分析
- **使用场景**：仪表盘数据分析、OLAP 查询、跨表聚合

### 2.3 对我们项目的启示

FxDB 和 WideQuery 是一个 **100+ 人基础设施团队**多年的成果。对于一个团队来说，直接使用 PostgreSQL（并充分利用其 JSONB/GIN/分区/并行查询能力）远比自研数据库现实。**不要试图模仿 FxDB。**

## 三、实时协作：OT + CRDT 混合架构

### 3.1 确认使用的技术

从多个技术来源交叉验证，飞书的协作系统是**混合架构**：

| 场景 | 协议 | 原因 |
|------|------|------|
| **飞书文档（富文本）** | 自研 OT | 文本操作高频细粒度，OT 在此场景更优 |
| **飞书多维表格** | **CRDT** | 表格冲突复杂（多单元格、行列增删、公式级联），CRDT 的交换律/结合律/幂等性保证最终一致性 |

### 3.2 Hyper Sync（2025 升级）

- 2024: 支持 100 人并发
- 2025: 支持 **1000 人**并发（10x 提升）
- 推测使用了 CRDT 数据结构 + 独占式服务端优化（批处理、分片、或改进的 CRDT 实现）

### 3.3 协作底层机制（推测）

基于飞书文档团队在 QCon 2022 的公开演讲：

1. **Vector Clock**：因果序追踪
2. **ChangeSet 三层模型**：A（服务端最新）、X（已提交未确认）、Y（本地未提交）
3. **OT `follow` 算法**：转化并发操作为一致序列（文档场景）
4. **Compose**：ChangeSet 定期压缩为 Snapshot
5. **WebSocket**：双向实时同步
6. **CRDT for 表格**：本地 CRDT 数据结构支持离线编辑，重连时自动合并

### 3.4 关键洞察

飞书是 **1000 人基础设施团队 + 字节跳动内部基建** 的产物。他们的协作方案包含了 OT、CRDT、自定义协议、自研同步算法的混合体。没有单一的开源方案可以等价代替。

对于我们的项目，核心问题是：**"50 人并发"和"1000 人并发"之间的架构差异是数量级的**。为 50 人做的方案（乐观锁或简单 CRDT）和为 1000 人做的方案（分布式 CRDT + 自研同步引擎）完全不同。不要在 MVP 阶段为 1000 人设计。

## 四、Rust 公式引擎

### 4.1 确认信息

- **Rust 实现**，属于 Lark SDK 核心（`liblark.dll`，约 100MB）
- 内部称呼：「Rust 公式迅算引擎」→「Rust 超维计算引擎」
- 2024: 万行复杂计算从 1-2 分钟 → **5 秒以内**
- 2025: 升级到**毫秒级**（千万行规模）

### 4.2 架构推测

```
服务端:
  公式计算 → Rust 原生引擎 → MPP/WideQuery 分发 → 批量向量化执行

客户端 (Web):
  SQLite (WASM) → 内存关系数据库 → Rust 公式引擎 (WASM) → Canvas 渲染

集成方式:
  Lark SDK 单体 DLL → 包含所有核心逻辑（公式/协作/数据）
```

### 4.3 WASM 实践

飞书在 2021 年就完成了 SQLite → WASM 的移植工作（RustMagazine 2021）：
- 目标平台：`wasm32-unknown-unknown`（零外部依赖）
- 自定义 Memory VFS（虚拟文件系统）
- 自定义内存分配器
- 客户端作为**纯内存数据库**运行，不做浏览器持久化
- 使用 **diesel** ORM 进行跨平台 SQLite 操作

### 4.4 对我们项目的启示

飞书用 Rust 做公式引擎是对的，但要认识到他们有一个专门的 Rust 团队 + 字节跳动的 Rust 内部生态（TiKV、RustRPC 等）。对于我们的项目，Formualizer 是更合理的起点——它已经提供了 320+ 公式和 WASM 支持，不需要从头写。

## 五、前端渲染引擎

### 5.1 三层渲染架构

| 层级 | 技术 | 适用场景 |
|------|------|---------|
| 基础层 | 虚拟 DOM/List | 基础表格视图、小数据集 |
| 标准层 | **自研 Canvas 渲染引擎** | 中大型表格（常规多维表格视图） |
| 极限层 | **Skia 编译为 WASM** | 亿级数据、复杂仪表盘 |

### 5.2 Widget 架构（Flutter 启发）

飞书多维表格的前端渲染使用自研 Widget 系统（受 Flutter 启发）：

- **状态驱动**：声明式 Widget，数据变更时 diff 更新
- **Flex 布局**：Column, Row, Flex, Expanded 等布局原语
- **虚拟列表 Widget**：支持嵌套虚拟化（虚拟列表中的虚拟列表）
- **Key-based Diffing**：最小化 DOM/Canvas 更新
- **双缓冲/离屏渲染**：保证滚动流畅
- **服务端 SSR + Canvas Hydrate**：Widget 树在服务端计算，客户端 Canvas 渲染（类似 React hydration）

### 5.3 性能数据

- 10 万行 × 100 列公式计算 < 5 秒（2024 Rust 引擎）
- 仪表盘可分析 1 亿+ 行、来自最多 200 张表的数据
- 千万行毫秒级计算（2025 升级）

### 5.4 微前端架构

飞书整体使用 **MPA + 微前端**架构：
- 文档、表格、多维表格是独立的微前端应用
- Monorepo **超 100 万行代码**
- pnpm + esbuild + CI/CD 统一流水线
- 前负责人张泽亚（QCon 广州 2022 演讲）

### 6.2 向量一体化存储

2025 新增核心引擎，**单次写入同时完成结构化存储 + 向量化处理**：
- 同时支持结构化查询（SQL-like）和向量相似度检索
- 为 RAG 应用提供原生数据底座
- 消除独立向量数据库（Milvus/Pinecone）的需求
- 被认为是飞书在数据库领域最前沿的技术

## 七、对我们的核心启示

### 7.1 什么可以学

| 飞书的选择 | 我们可以借鉴 |
|-----------|-------------|
| CRDT for 表格（非纯 OT） | 确认了 CRDT 是多维表格的正确方向 |
| Rust 公式引擎 + WASM | Formualizer 方案与飞书技术路线一致 |
| Widget 渲染架构 | Canvas 渲染 + Widget 树概念 |
| 微前端 MPA | 初期不需要，但架构预留扩展点 |
| 分层存储（热/温/冷） | PostgreSQL 分区 + LRU 内存缓存可以实现 |

### 7.2 什么不可学

| 飞书的选择 | 为什么不可学 |
|-----------|-------------|
| FxDB 自研数据库 | 需要 100+ 人基础设施团队，数年开发 |
| WideQuery MPP 引擎 | 同上，成本极高 |
| 1000 人并发协作 | MVP 不需要，为 50 人设计即可 |
| Hyper Sync 自研协议 | 不如直接用成熟 CRDT 库 |
| 向量一体化存储 | 极其前沿，V1 用独立方案即可 |
| 1 亿行仪表盘 | MVP 不需要，PostgreSQL 处理 10 万行足够 |

### 7.3 关键数字锚定

飞书从 0 → 1000 万行的演进路径告诉我们：

| 阶段 | 技术选择 | 飞书用了多久 |
|------|---------|------------|
| 1 万行以下 | PostgreSQL 单机足够 | 初期 |
| 1 万 ~ 100 万行 | PostgreSQL 优化 + Rust 引擎 + CRDT | 2024 年达到 |
| 100 万 ~ 1000 万行 | 自研 FxDB + MPP + 分层存储 | 2025 年达到 |

**我们的 V1 目标（10 万行）在飞书的第一个阶段**——这意味着 PostgreSQL 单机 + 基础 CRDT 或乐观锁完全够用，不需要自研数据库。
