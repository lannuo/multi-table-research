# 决策分析：Fork 开源项目 vs 从零构建

> 日期: 2026-04-24
> 背景: 全功能多维表格产品，solo 开发，VIBE CODING 模式
>
> **许可约束**: 仅使用 MIT/Apache-2.0 许可的项目。APITable 和 Teable 均为 AGPL-3.0，Fork 后衍生作品也须以 AGPL 许可发布。这对 VIBE CODING 的 solo 开发者是额外的法律负担——即使技术上可行，许可证约束也使 Fork 方案不具备可行性。

---

## 一、候选方案

### 方案 A：Fork APITable（AGPL-3.0）

**APITable 已有的东西**：

| 模块 | 状态 | 技术 |
|------|------|------|
| Canvas 渲染引擎 | 成熟 | Konva.js 自研层 |
| 实时协作 | 成熟 | OT (Server-centric) |
| 公式引擎 | 成熟 | TypeScript 实现 |
| 字段类型 (10+) | 成熟 | JSONB 动态字段 |
| 视图 (6 种) | 成熟 | Grid/Kanban/Gallery/Calendar/Gantt/Form |
| 权限系统 | 成熟 | Mirror + 列权限 |
| 插件/Widget 系统 | 成熟 | iframe 沙箱 + SDK |
| 导入导出 | 成熟 | Excel/CSV |
| API | 成熟 | RESTful + WebSocket |
| 自动化 | 成熟 | Robot (Trigger→Action) |

**需要改的东西**：

| 需要改 | 原因 |
|--------|------|
| OT → CRDT | 我们确认 CRDT 是正确方向 |
| Java Spring Boot 后端 | 我们需要单语言栈 |
| MySQL → PostgreSQL | APITable 有两个后端，Java 部分用 MySQL |
| AGPL 许可证 | 如果 SaaS 部署无需开源，私有化部署需要 |
| 剥离不需要的模块 | 减少维护范围 |

### 方案 B：Fork Teable（AGPL-3.0）

**Teable 已有的东西**：

| 模块 | 状态 | 技术 |
|------|------|------|
| Canvas 表格渲染 | 成熟 | 自研 Canvas 组件 |
| 公式引擎 | 成熟 | ANTLR4 → SQL |
| 字段类型 | 成熟 | PostgreSQL 物理列 |
| 实时协作 | 成熟 | WebSocket + OT-like |
| 权限 | 成熟 | RBAC |
| 后端 | NestJS | ✅ 与我们偏好一致 |
| 数据库 | PostgreSQL | ✅ 与我们偏好一致 |

**需要改的东西**：

| 需要改 | 原因 |
|--------|------|
| OT-like → CRDT | 同 APITable |
| 物理表 → JSONB 混合 | Teable 的物理表对 Schema 灵活性限制大 |
| 公式 SQL → WASM 引擎 | Teable 的 SQL 公式表达能力有限（78 函数） |
| AGPL 许可证 | 同 APITable |

### 方案 C：从零构建

没有任何遗留代码，但有完整的选型自由。

---

## 二、核心约束：Solo 开发者

这个约束决定了 Fork 的大部分优势是伪优势：

| Fork 的假设优势 | 实际现实 |
|---------------|---------|
| "已有全部功能，省开发时间" | 理解一个 100K+ 行的代码库需要数周到数月。期间无法添加新功能 |
| "改改就能用" | APITable 要改 OT→CRDT、Java→TS、MySQL→PG — 是**大手术**，不是改配置 |
| "社区维护" | 上游更新后需要反复合并，合并冲突是常态 |
| "有参考实现" | 可以读源码学习，不需要 fork |

**Teable 相对好很多**——它只有一个语言栈（TS），用 PostgreSQL，架构更干净。但核心矛盾依旧：理解代码 > 修改代码 > 写新代码。

---

## 三、维度对比

| 维度 | Fork APITable | Fork Teable | 从零构建 |
|------|-------------|------------|---------|
| 首月产出 | 跑起来 demo，不理解内部 | 跑起来 demo，能改表层 | Canvas PoC + 数据模型 |
| 第三月产出 | 完成 OT→CRDT 改造 | 完成 OT→CRDT 改造 | Canvas 表格 + CRDT 同步 + CRUD |
| 技术栈控制 | 受限于原项目 | 受限于原项目 | 完全自由 |
| 代码理解度 | 表面理解 | 中等理解 | 深度理解每行 |
| 技术债 | 继承原项目全部 | 继承原项目全部 | 只有自己创建的 |
| 许可证风险 | AGPL（SaaS OK，私有化需要开源） | 同 | 自有代码，MIT 组件 |
| 公式引擎改造难度 | 极高（TS 引擎，架构耦合） | 中（SQL 方案，改 WASM 需重写） | 低（从零集成） |
| CRDT 改造难度 | 极高（OT 渗透整个系统） | 高（WebSocket 层需重写） | 低（从零集成 Loro） |

## 四、一个关键计算

理解一个代码库的时间 ≈ 写同样功能时间的 30-50%（如果代码质量好且有文档）。

APITable 粗略估计有 200K+ 行有效代码（TypeScript + Java）。理解它需要约 3-6 个月全职投入。

从零构建核心功能的代码量估计：

| 模块 | 预估代码行 | 说明 |
|------|-----------|------|
| Canvas 表格渲染 | 3,000-5,000 | PoC 已验证可行，~400 行 |
| CRDT 同步 (Loro) | 1,000-2,000 | loro-extended 提供大部分 |
| 数据模型 CRUD | 2,000-3,000 | RESTful API + JSONB |
| 字段类型系统 | 2,000-3,000 | 12 种字段类型 |
| 视图系统 | 3,000-5,000 | Grid + 表单初期 |
| 权限系统 | 2,000-3,000 | RBAC 基础 |
| 用户/认证 | 1,000-2,000 | JWT + 基本用户管理 |
| 导入/导出 | 1,000-2,000 | CSV/Excel |
| **合计** | **15,000-25,000** | — |

对比 Fork 的学习成本（理解100K+行）和改造工作量，对于 solo 开发来说，**从零写的总投入可能更低**。

而且从零写有另一个收益：**你会理解每一行代码的用途**。这是在长期持续开发中最重要的资产。

---

## 五、结论

**从零构建。**

### 理由

1. **Solo 开发的核心约束不是"缺代码"，而是"不理解代码"**。Fork 获得的代码你不理解，等于没有获得
2. **APITable 和 Teable 都需要架构大手术**（OT→CRDT，公式引擎更换），这不是浅层修改
3. **Teable 比 APITable 好很多，但它的物理表方案和公式→SQL 方案限制了全功能产品的公式表达能力**
4. **从零构建的实际代码量估算为 15K-25K 行**——对全功能产品来说是合理的范围
5. **我们已经验证了最不确定的部分**（Canvas 渲染 PoC 1）
6. **Loro + loro-extended 提供了大量的现成模块**（CRDT 核心、React Hooks、PG 适配器、IndexedDB 离线）

### 不排斥的

- **读 APITable 和 Teable 的源码来理解特定功能的实现**（Canvas 事件处理、字段类型定义、导入导出等）
- **复用 loro-extended 等现成库**——这不叫 fork，叫用依赖
- **遇到特别复杂的模块（如 Excel 导入），可以考虑引入现成库而不是自己写**

### 风险

- 全功能产品范围大，solo 开发周期长
- 需要严格管理范围，按优先级增量交付
- 建议的构建顺序：Canvas 表格 → 字段类型 → CRDT 同步 → 权限 → 视图 → 导入导出 → 公式引擎 → 自动化
